What is Algorithmic Anaylsis Anyways

We will be learning about algorithmic analysis. Algorithmic analysis can be defined as the process of finding the computational complexity of algorithms both in time, storage, or other resources needed to execute them.

Complexity Theory

For now we are first going to take a look at the time complexity aspect of an algorithm. For time, and all complexities, there is a general relationship associated with an algorithms design and it complexities. Complexities, not in the sense of understanding the coder, but in the algorithm itself. This relationship is focused more between the length of the input for the algorithm and the resouces used by the algorithm. Thus, the resouces needed for the algorithm can vary depending on the input size and type. Most obviously, the greater the size of the input the greater the amount of resouces needed. What can be less obvious is the effect the input type has on the resources required. We will take a greater look into this relationship a scroll down.

Best Case Time Complexity - \(\Omega\)

We will take a look at a well know sorting algorithm in order to get out feet wet. This algorithm is insertion sort. Insertion sort can be defined by the following psudeo-code.

pseudocode: Insertion Sort Algorithm function Insertion Sort(List L) i = 0 while i < length(L) j = i while j > i and L[j-1] > L[j] swap L[j] and L[j-1] j = j - 1 end while i = i + 1 end while

If this pseudo-code doesn't make all that much sense, that is fine of course. However, in order to continue with the rest of this it is best you understand it. Here is a quality website's explanation and in depth look at insertion sort

Insertion sort is a great introduction algorithm to look at and understand the effect the type of input has on the time complexity of the algorithm. First, think what would happen if all the data in the input, or the type of input, was already sorted in ascending order. Emphasis on ascending. Although unrealistic, it is a good to see its effects. Well, based on the way insertion sort operates, the algorithm would not have to make any changes. Really, it could traverse through the whole list in one loop. Meaning the code in the main loop would only have to execute once for every item in the list, or the size of the list. The size of the list is denoted as \(n\).

This is called a best-case time complexity. Why, well because best-case time complexity can be summarized by saying the minimum number of steps an algorithm needs to take on any instance of size n. This will be the least amount of steps an algorithm could theoretically operate in. In essence, it is a lower bounds. This is best-case time complexity is denoted with \(\Omega\), a.k.a. Big-Omega.

Now, we need to look at this idea of lower bounds. In essesence, a lower bound describes an element of some function for which is is less than or equal to every element of that function. Basically, a limit for which nothing can pass. Upper bound, meaning nothing can go higher than y1, and lower bound meaning nothing can go lower than y2.

Just like there exists a lower bounds, our best-case time complexity, there also exists an upper-bound, and a tight-bound. All of which will be discussed in the later sections.

Best-Case Time Complexity Equation

As described above, best-case time complexity can be descirbed as the minimum number of steps an algorithm needs to take on any instance of size n. Mathematically, it is described by the following equation:

$$\Omega(g(n)) = \exists \, c > 0, n_{0} > 0$$
$$s.t. \; 0 \leq cg(n) \leq f(n)$$
$$\forall \, n \geq n _{0}$$

Lets break this down just in case you're not too familiar with the notation. Some of this is axiomatic, but I'll explain nevertheless. I will break this down into individual parts, then piece them together to arrive at the final equation. First, lets start with what's left of the equal sign.

$$\Omega(g(n))$$

\(\Omega\) is Big-Omega which denotes best-case time complexity or the lower-bounds. \(g(n)\) is describing the function of \(g\) on \(n\), a.k.a. the input size. Next, lets look at

$$\exists \, c > 0, n_{0} > 0$$

This is saying that there exists two constants, \(c\) and \(n_{0}\). \(c\) accounts for things like runtime constants such as processor speed, memory speed, etc.. It also accounts for other runtime constants in the code such as conditional statements, variable initiliazations, etc. that are asymptotically held constant. Now, these constants must be above 0.

$$0 \leq cg(n) \leq f(n)$$

Then there is \(0 \leq cg(n) \leq f(n)\). This means that \(cg(n)\) lies between \(0\) and \(f(n)\). In this instance, \(f(n)\) is the original asymptotic function, the one that is derived from the algorithm itself. \(cg(n)\) denotes the lower-bounds of our original asymptotic function. Thus, this is saying that the lower-bounds of the original asymptotic function for our algorithm lies at or below the original asymptotic function or at or above \(0\).

$$\exists \, c > 0, n_{0} > 0$$
$$s.t. \; 0 \leq cg(n) \leq f(n)$$

Now, to concatenate the last two parts together. This segment of the equation is saying that there exists two constants \(c\) and \(n_{0}\) such that our lower-bounds of the original asymptotic function \(cg(n)\) lies at or below the original asymptotic function \(f(n)\) or at or above \(0\).

So putting it all together and we have the equation:

$$\Omega(g(n)) = \exists \, c > 0, n_{0} > 0$$
$$s.t. \; 0 \leq cg(n) \leq f(n)$$
$$\forall \, n \geq n _{0}$$

From what we have learned we can say that this equation here says that there exists a constant \(c\) and \(n\), \(c\) being runtime constants, that are greater than 0 such that our lower-bound exists between 0 and the original asymptotic function.
